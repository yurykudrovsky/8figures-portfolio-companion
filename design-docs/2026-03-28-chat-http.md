# Design Doc — Chat HTTP Integration — 2026-03-28

## Problem Statement

`ChatService.sendMessage()` currently calls `this.buildResponse()`, a private method that pattern-matches the user's message against a fixed set of keywords and returns a locally-assembled string. No HTTP request is ever issued. The backend `POST /api/chat` endpoint — which proxies to the real Claude API when `ANTHROPIC_API_KEY` is present — is never reached. The iOS Simulator therefore never sees a real AI response.

This design addresses the functional gap directly. It also resolves audit finding **W5** (magic number `15` in `chat.service.ts` — the service currently uses an inline `}, 15)` rather than the named constant that `STREAMING_INTERVAL_MS` should provide).

No other audit findings are in scope for this task.

---

## Proposed Solution

Rewrite `ChatService` to use `HttpClient` for the `sendMessage()` path, following the exact DI and platform-detection pattern already established by `PortfolioService`. The raw SSE body returned from `POST /api/chat` (a `text/event-stream` response read in full as a single `responseType: 'text'` string) is parsed by a private `parseSseBody()` helper into a plain text string. That assembled text is then re-streamed character-by-character through the existing `streamResponse()` method. When the backend is unreachable, `catchError` streams a static fallback string via the same `streamResponse()` path — the Observable never errors to the subscriber. All routing logic (`buildResponse` and its five sub-methods) and all formatting helpers (`fmt`, `fmtChange`, `fmtPct`, `matches`) are removed because Claude now owns that responsibility.

This approach is minimal: one new dependency (`HttpClient`), one new private method (`parseSseBody`), and a rewritten `sendMessage()`. The public API surface — `sendMessage()` and `streamResponse()` — is unchanged in signature, so the component layer requires zero modification.

## Interface Definitions

Internal types used only within `chat.service.ts`. Neither interface is exported.

```typescript
// Shape of a single SSE frame produced by the backend.
// The backend emits exactly one of: char present, done flag, or error message.
interface SseCharFrame {
  char: string;
}

interface SseDoneFrame {
  done: true;
}

interface SseErrorFrame {
  error: string;
}

// Union of all frame shapes the parser must handle.
type SseFrame = SseCharFrame | SseDoneFrame | SseErrorFrame;
```

Type guards BUILDER must implement (these are shape contracts, not code):

- `isCharFrame(frame: SseFrame): frame is SseCharFrame` — returns true when `'char' in frame`
- `isDoneFrame(frame: SseFrame): frame is SseDoneFrame` — returns true when `'done' in frame`

## API Contract

### POST /api/chat (existing — no changes required to backend)

**Request**

```typescript
interface ChatPostRequest {
  message: string;
  context: {
    portfolio: Portfolio;
    messages: ChatMessage[];
  };
}
```

**Response** — `Content-Type: text/event-stream`, read as full text body

Each SSE block is separated by `\n\n`. Possible block forms:

```
data: {"char":"H"}\n\n
data: {"char":"i"}\n\n
data: {"done":true}\n\n
data: {"error":"some message"}\n\n
```

**Error responses (backend validation)**

| Status | Body |
|--------|------|
| 400    | `{ "error": "message is required" }` |
| 400    | `{ "error": "portfolio context is required" }` |

The frontend treats any non-2xx HTTP response as the error path and streams the static fallback string.

## sendMessage() Behaviour Specification

1. Construct the request body as `ChatPostRequest` (defined above).
2. Call `this.http.post<string>(${this.apiUrl}/chat, body, { responseType: 'text' as const })`.
3. Pipe through `map(rawBody => this.parseSseBody(rawBody))` to obtain the assembled text.
4. Pipe through `switchMap(text => this.streamResponse(text))` to re-stream character by character.
5. Pipe through `catchError(_ => this.streamResponse(FALLBACK_MESSAGE))` where `FALLBACK_MESSAGE` is a module-level constant holding a user-facing apology string such as `"I'm having trouble reaching the server right now. Please try again in a moment."`.
6. Return the resulting `Observable<string>` directly — no `.subscribe()` inside the service.

The `history` parameter is passed as `context.messages` in the request body. It must not be renamed `_history` (the underscore prefix was a lint-suppression for the unused parameter; once the parameter is used it must be named plainly).

## parseSseBody() Behaviour Specification

Input: a raw string body from the SSE endpoint, e.g.:

```
data: {"char":"H"}\n\ndata: {"char":"i"}\n\ndata: {"done":true}\n\n
```

Steps:

1. Split the input string on `'\n\n'`.
2. Filter blocks to those that begin with `'data: '`.
3. For each qualifying block, strip the `'data: '` prefix and `JSON.parse` the remainder into an `SseFrame`.
4. If the frame satisfies `isCharFrame`, append `frame.char` to the accumulator string.
5. If the frame satisfies `isDoneFrame` or `isSseErrorFrame`, skip it (do not accumulate).
6. Return the accumulated string.

The method must be resilient to malformed JSON in any single block (a try/catch per block that skips unparseable frames is sufficient; do not let one bad frame throw from the method).

## Test Structure BUILDER Must Match

The new spec file replaces all existing tests. The 10 `streamResponse()` tests carry over verbatim (no changes). The routing and content tests (17 tests) are removed and replaced with 7 HTTP-layer tests. Final count: **17 tests** (10 existing + 7 new).

### Outer structure

```
describe('ChatService')
  beforeEach  → vi.useFakeTimers() + TestBed.configureTestingModule(...)
  afterEach   → httpMock.verify() + vi.useRealTimers()

  describe('streamResponse()')         — 10 tests (carry over unchanged)

  describe('sendMessage() — HTTP layer')  — 7 tests (new)
```

### TestBed configuration

```
TestBed.configureTestingModule({
  providers: [
    ChatService,
    provideHttpClient(),
    provideHttpClientTesting(),
  ],
})
```

`ChatService` must be obtained via `TestBed.inject(ChatService)`, not `new ChatService()`.

### Capacitor mock

`ChatService` imports `Capacitor` from `@capacitor/core` (same as `PortfolioService`). The spec must hoist a `vi.mock('@capacitor/core', ...)` identical to the one in `portfolio.service.spec.ts`, and set `isNativePlatform → false`, `getPlatform → 'web'` in `beforeEach`.

### collectStream() helper rewrite

The existing `collectStream()` helper calls `service.sendMessage()` and runs timers to collect characters. In the new spec, `sendMessage()` requires an HTTP request to be flushed before timers are advanced. The helper must be replaced with an inline pattern inside each HTTP test: flush the request, then call `vi.runAllTimers()`, then assert on collected chars.

### The 7 new sendMessage() HTTP tests

| # | Test name | What to assert |
|---|-----------|---------------|
| 1 | `makes POST to /api/chat` | `httpMock.expectOne('http://localhost:3000/api/chat')`, method is `'POST'` |
| 2 | `request body contains message and portfolio context` | `req.request.body.message === 'test question'`; `req.request.body.context.portfolio.id === 'p1'` |
| 3 | `request body includes message history` | `req.request.body.context.messages` is an array of length matching the history passed in |

| 5 | `ignores done frame — no extra character` | Flush with done-only SSE; after timers collected chars is empty string |

`parseSseBody` is private. It is covered indirectly by tests 4 and 5 above. No separate `describe('parseSseBody()')` block is required.

## Open Questions

All questions have been resolved by codebase inspection.

**Q1: Does `HttpClient` need to be provided at the app level, or does TestBed need it?**
Resolved: `provideHttpClient()` is already wired in `app.config.ts` (used by `PortfolioService`). No app-level change needed. The spec must add `provideHttpClient()` + `provideHttpClientTesting()` to its own `configureTestingModule` providers, matching the pattern in `portfolio.service.spec.ts`.

**Q2: Does the `responseType: 'text'` option interact with Angular's type inference in a way that requires a cast?**
Resolved: Yes. `this.http.post(url, body, { responseType: 'text' as const })` — the `as const` cast is required so TypeScript narrows `responseType` to the literal `'text'` rather than `string`. This is an established Angular pattern; BUILDER must use it.

**Q3: Is there a `environment.ts` for the `apiUrl` base?**
Resolved: `frontend/src/environments/environment.ts` exports `{ production: false, apiUrl: 'http://localhost:3000/api' }`. The `apiUrl` getter must use `environment.apiUrl` as the base, then apply the Android `10.0.2.2` substitution identically to `PortfolioService`.

**Q4: Should `history` be the full `ChatMessage[]` array passed as `context.messages`, or filtered?**
Resolved: Pass the full array as received. The backend accepts `context.messages` as `ChatMessage[]` (see `backend/src/models/portfolio.model.ts` — `ChatRequest.context.messages`). No filtering is required in the service layer.

**Q5: What SSE error frame format does the backend emit?**
Resolved: `data: {"error":"<message>"}\n\n` (see `backend/src/routes/chat.ts:148`). `parseSseBody` must skip these frames (do not accumulate the error string as content). The error path at the HTTP layer handles total request failure; the `error` frame within SSE is a within-stream signal that is silently skipped.

**Q6: Does Vitest's fake timer setup need to change given the addition of `TestBed`?**
Resolved: No structural change. `vi.useFakeTimers()` is called in the outer `beforeEach` before `TestBed.configureTestingModule` — `TestBed` setup is synchronous and does not require real timers. The existing ordering (`vi.useFakeTimers()` first, then TestBed, then inject) is correct.

**Q7: Does W5 (the magic number finding) require the `STREAMING_INTERVAL_MS` constant to be referenced in the `setInterval` call?**
Resolved: Yes. The current file declares `const STREAMING_INTERVAL_MS = 15` at module scope but the `setInterval` at line 20 passes `15` directly (see `chat.service.ts:20`). BUILDER must change that call site to use `STREAMING_INTERVAL_MS`. This is a one-line fix bundled into the same file change.
